## 🚀 기여 방법

### 버그 제보

버그를 발견하면 다음 내용을 포함하여 GitHub에 이슈를 열어주세요:

- 문제에 대한 명확한 설명
- 재현 단계
- 예상 결과 vs 실제 결과
- 환경 정보 (OS, Rust 버전, DBX 버전)

### 기능 제안

새로운 기능 제안을 환영합니다! 다음 내용을 포함하여 이슈를 열어주세요:

- 기능에 대한 명확한 설명
- 사용 사례 및 이점
- 구현 아이디어 (선택 사항)

### 코드 기여

1. GitHub에서 **레포지토리를 포크(Fork)**합니다.
2. **기능 브랜치를 생성**합니다 (`git checkout -b feature/my-feature`).
3. 코딩 표준을 준수하며 **변경 사항을 적용**합니다.
4. 변경 사항에 대한 **테스트를 추가**합니다.
5. **테스트를 실행**합니다 (`cargo test --workspace`).
6. 커밋 메시지 형식을 준수하여 **변경 사항을 커밋**합니다.
7. 본인의 포크로 **푸시(Push)**합니다 (`git push origin feature/my-feature`).
8. GitHub에서 **풀 리퀘스트(Pull Request)를 생성**합니다.

## 🛠️ 개발 환경 설정

### 필수 요구 사항

- Rust 2024 에디션 이상
- CUDA 12.x+ (선택 사항, GPU 기능 사용 시)

### 빌드 및 실행

```bash
# 레포지토리 클론
git clone https://github.com/ByteLogicStudio/DBX.git
cd DBX

# 프로젝트 빌드
cargo build --workspace

# 테스트 실행
cargo test --workspace

# 벤치마크 실행
cd testing/benchmarks
cargo bench
```

## 📝 기여자 라이선스 동의서 (CLA)

**DBX에 기여함으로써 귀하는 다음 약관에 동의하게 됩니다:**

### 저작권 라이선스 부여

귀하는 ByteLogic Studio 및 ByteLogic Studio가 배포하는 소프트웨어 수신자에게 귀하의 기여물 및 그 파생물에 대해 다음을 수행할 수 있는 영구적이고 전 세계적이며 비독점적이고 무상이며 로열티가 없고 취소 불가능한 저작권 라이선스를 부여합니다:

- 복제, 파생물 제작, 공개 전시, 공개 시연, 서브라이선스 부여 및 배포.

### 특허 라이선스 부여

귀하는 ByteLogic Studio 및 ByteLogic Studio가 배포하는 소프트웨어 수신자에게 귀하의 기여물에 대해 다음을 수행할 수 있는 영구적이고 전 세계적이며 비독점적이고 무상이며 로열티가 없고 취소 불가능한 특허 라이선스를 부여합니다:

- 제조, 제조 의뢰, 사용, 판매 제안, 판매, 수입 및 기타 양도.

### CLA가 필요한 이유

DBX는 **이중 라이선스 모델**(오픈 소스 MIT + 상업용 라이선스)을 사용합니다. 상업용 라이선스를 제공하기 위해 ByteLogic Studio는 기여물에 대한 서브라이선스 부여 권한이 법적으로 필요합니다.

**귀하의 기여는 오픈 소스 커뮤니티를 위해 항상 MIT 라이선스 하에 자유롭게 제공됩니다.** CLA는 우리가 프로젝트 개발 자금을 마련하기 위해 상업용 라이선스를 함께 제공할 수 있도록 허용하는 장치일 뿐입니다.

### 대리권 보장

귀하는 다음과 같이 진술합니다:

- 귀하는 상기 라이선스를 부여할 법적 권한이 있습니다.
- 귀하의 기여물은 귀하의 독창적인 창작물입니다.
- 귀하의 기여물이 제3자의 권리를 침해하지 않습니다.

### CLA 서명

풀 리퀘스트를 제출함으로써 귀하는 본 CLA를 읽고 이에 동의함을 인정하게 됩니다. 기업 기여자의 경우 별도의 기업용 CLA 서명이 필요하므로 license@bytelogic.studio로 문의해 주세요.

## 🔄 풀 리퀘스트(PR) 프로세스

1. **모든 테스트 통과 확인**
   ```bash
   cargo test --workspace
   cargo clippy --workspace -- -D warnings
   cargo fmt --check
   ```
2. 새로운 기능을 추가하는 경우 **문서 업데이트**
3. 새로운 기능에 대한 **테스트 케이스 추가**
4. **커밋 메시지 형식 준수** (아래 참조)
5. **리뷰 대기** - 가능한 한 빨리 PR을 검토하겠습니다.
6. **피드백 반영** - 요청된 변경 사항을 적용하고 업데이트를 푸시합니다.
7. **머지(Merge)** - 승인되면 PR이 머지됩니다.

## 💻 코딩 표준

### Rust 스타일 가이드

- [Rust API 가이드라인](https://rust-lang.github.io/api-guidelines/)을 따릅니다.
- 포맷팅을 위해 `cargo fmt`를 사용합니다.
- 린팅을 위해 `cargo clippy`를 사용합니다.
- 관용적인(Idiomatic) Rust 코드를 작성합니다.

### 커밋 메시지 형식

우리는 [Conventional Commits](https://www.conventionalcommits.org/) 사양을 따릅니다:

```
[유형]: 요약 설명

상세 설명 (선택 사항)

Fixes #123
```

**유형:**
- `[기능]`: 새로운 기능 추가
- `[수정]`: 버그 수정
- `[문서]`: 문서 변경
- `[스타일]`: 코드 스타일 변경 (포맷팅 등)
- `[리팩토링]`: 코드 리팩토링
- `[테스트]`: 테스트 추가 및 업데이트
- `[빌드]`: 빌드 시스템 변경
- `[CI]`: CI/CD 설정 변경
